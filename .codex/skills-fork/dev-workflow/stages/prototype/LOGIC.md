# Logic Prototype

一个可交互的小终端 app，让用户手动驱动状态模型。问题关于**业务逻辑、状态转换、数据形状**时用——纸上看着合理，推过真实 case 才觉得不对的那种。

## 何时选这条分支

- "我不确定这状态机处理不处理 X 然后 Y 的 edge case。"
- "这数据模型真让我表达……的情况吗？"
- "我想在写 API 前摸出它该长什么样。"
- 任何用户想**按按钮看状态变**的情况。

若问题是"该长什么样"——走错分支。用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

写代码前，写下要原型什么状态模型 + 什么问题。一段话，放原型 README 或文件顶部注释。回答错问题的逻辑原型是纯浪费——把问题写明，方便事后核对。

### 2. 选语言

优先用宿主项目已有 runtime。项目无明显 runtime（如 docs repo）时，先检查环境已有工具，模型选择一个无需安装、最贴近问题的 runtime 并记录理由；不要把语言偏好问给用户，也不要为原型引入新包管理器。只有 runtime 会实质改变待验证行为或交付约束、且该 node 本来就是 `owner:user` 时，才把它作为当前同一个 decision node 交给 grilling；没有可用工具则带证据返回 blocked。

### 3. 把逻辑隔离进可移植模块

把真正回答问题的逻辑——放小而纯的接口后，能日后 lift 出塞进真实代码库。围绕它的 TUI 是 throwaway；逻辑模块不该是。

合适形状看问题：

- **纯 reducer** — `(state, action) => state`。动作是离散事件、状态是单值时好用。
- **状态机** — 显式状态 + 转换。"哪些动作此刻合法"是问题一部分时好用。
- **纯函数集** over 纯数据类型。无隐式当前状态、只变换时好用。
- **类/模块** 有清晰方法面，当逻辑真拥有持续内部状态时。

选最贴合问题的形状，*不*选最易接 TUI 的。保持纯：无 I/O、无终端代码、无控制流 `console.log`。TUI import 它调它；不反向流。

这让原型超出自身寿命：问题答完，验证过的 reducer/机器/函数集能单独 lift 进真实模块。

### 4. 建最小 TUI 暴露状态

建**轻量 TUI**——每 tick 清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价）重渲染整帧。用户该始终见一个稳定视图，不是不断增长的 scrollback。

每帧两部分，按序：

1. **当前状态**，pretty-print 且 diff 友好（一行一字段或格式化 JSON）。字段名/标题用**粗体**，次要上下文（时间戳/ID/派生值）用**暗淡**。原生 ANSI 转义码就行——`\x1b[1m` 粗体、`\x1b[2m` 暗、`\x1b[0m` 重置。无需引样式库，除非项目已有。
2. **键盘快捷键**，列底部：`[a] 加用户  [d] 删用户  [t] 推进时钟  [q] 退出`。粗体 key 暗描述，或反之——读着干净就行。

行为：

1. **初始化状态** — 单内存对象/struct。启动渲染首帧。
2. **读一键**（或一行），dispatch 到 mutate 状态的 handler。
3. **每次动作后重渲染整帧** — 不 append，replace。
4. **循环到退出。**

整帧该一屏装下。

### 5. 一条命令跑起来

加脚本到项目现有 task runner（`package.json` scripts / `Makefile` / `justfile` / `pyproject.toml`）。用户跑 `pnpm run <prototype-name>` 或等价——绝不该记路径。

宿主项目无 task runner 就把命令放原型 README 顶部。

### 6. 交付

给运行命令。用户自己驱动；有趣时刻是他们说"等等这不该可能"或"嗯我以为 X 会不同"——那是_想法_里的 bug，正是重点。要加新动作就加。原型会演进。

### 7. 捕获答案和原型

原型答完问题，捕获答案，再按 [STAGE](STAGE.md) 描述返回父决策 delta。逻辑专属映射：验证过的 reducer/机器/函数集 lift 进真实模块（决策吸收）；TUI 壳随 throwaway 分支保留原型作原始资料。

## Anti-patterns

- **别加测试。** 需要测试的原型不再是原型。
- **别接真实数据库。** 用内存存储，除非问题专门关于持久化。
- **别泛化。** 别"万一以后要支持 X"。原型答一个问题。
- **别把逻辑和 TUI 混一起。** reducer/状态机引用 `console.log`、prompt、终端转义码就不再可移植。TUI 保持纯模块上的薄壳。
- **别把 TUI 壳发生产。** 壳为终端手动驱动优化。背后的逻辑模块才值得留。
