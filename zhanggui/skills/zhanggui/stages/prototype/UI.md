# UI Prototype

用浏览器中的可切换变体回答一个视觉或交互设计问题。默认 2-3 个结构性不同的变体，最多 5 个——问题空间只有两个真实方向时 2 个即可，不硬造第三个；用户可以选一个、组合部件或否决全部。

## 进入条件

- 当前 UI 模式是 `Prototype-Assisted`。
- `Design-Owner` 流程中有一个问题靠文字无法可靠判断。
- 用户直接要求 mockup、HTML 原型、布局对比或交互演示。

用户选择 `Text-Only`、已有明确设计稿，或只是微调颜色/文案时不进入本分支。

## 第 1 步：写清问题

在文件顶部注释中写：

```text
PARENT DECISION ID: [Dn]
PROTOTYPE QUESTION: [需要通过视觉比较回答的问题]
SUCCESS SIGNAL: [用户看完后能够做出的决定]
```

没有 parent id 或 success signal 就不要开始写原型。

## 第 2 步：选择宿主

### A. 集成式原型

仅当现有页面的真实导航、sidebar、数据密度、权限或响应式容器会实质改变判断时使用。

- 使用项目现有框架和设计系统。
- 在 dev-only 路由、feature flag 或受控 query 参数下挂变体。
- 保留只读数据获取；所有 mutation 指向 stub 或隔离环境。
- 使用构建环境的开发标志隐藏原型入口，不能只检查 hostname。

### B. 独立 HTML

无合适宿主、greenfield、文档项目，或用户明确想快速双击查看时使用。

- 单个 HTML，内嵌 CSS 和 JS，无构建依赖。
- `file://` 与本地静态服务器都能工作。
- 原型切换器始终显示；不存在生产泄漏问题。
- 用真实感内容和稳定尺寸模拟最终密度，不做空壳卡片墙。

A 与 B 是有条件的选择，不是两个同时生效的默认。

## 第 3 步：必要时收集视觉参考

新产品、陌生领域或缺少现有设计系统时，查看 2–3 个有代表性的同类产品或官方设计指南。提取信息层级、导航、主要操作和密度等可复用原则，不复制品牌外观。

## 第 4 步：生成结构性变体

每个变体必须在至少一项上真正不同：

- 布局和空间组织。
- 信息层级与扫描顺序。
- 主要操作和导航模型。
- 工作流步骤或状态呈现。

只换颜色、圆角、字体或文案不算不同变体。两个方案太像时重做其中一个。

遵循宿主项目的视觉语言和受众：

- 操作工具优先安静、紧凑、易扫描，不做营销 hero。
- 不嵌套卡片，不用装饰性渐变球或单一色相铺满界面。
- 使用现有图标库；有 lucide 时不手画 SVG。
- 控件尺寸稳定，长文本不溢出，桌面和移动端不重叠。
- 交互状态完整：默认、hover/focus、空、加载、错误和禁用状态按问题需要呈现。

## 第 5 步：切换器

使用 `?variant=A|B|C` 保存当前变体，并提供底部浮动切换器：

- 左右箭头循环切换。
- 显示 key 和短名称，例如 `B — 左侧工作流`。
- `history.replaceState` 更新 URL，刷新后保持。
- 键盘左右键切换；输入框、textarea 或 contenteditable 聚焦时不拦截。

集成式原型用框架原生组件；独立 HTML 用共享 JS。不要把整个页面实现重复三份后再靠字符串拼接维护。

## 第 6 步：实际验证

交给用户前必须实际打开：

1. 逐个切换所有变体，刷新并直接访问各 `?variant=` URL。
2. 验证关键按钮、菜单、tab、表单和状态变化。
3. 检查桌面与移动视口，没有遮挡、溢出、空白画布或布局跳动。
4. 检查控制台和资源加载错误。
5. 集成式原型确认生产构建不会暴露入口或切换器。

这是一项 UI smoke check，不要求为 throwaway 原型建立自动化测试。

## 第 7 步：交付和退出

交付文件路径或 URL，说明每个变体的核心差异，并按入口"提问格式"标注推荐变体与理由、保留自由输入出口——选一个、组合部件、描述新方向或全部否决都是合法回答；描述新方向时按收敛循环迭代变体。用户反馈后记录：

```text
ParentDecisionId: Dn
Result: [胜出方向 A/B/C/组合/全部否决 + 决定依据 + 从其他变体借用的部件]
Rejected: [不采用的方向和原因]
PrototypeStatus: pending | answered
StageStatus: awaiting-user | node-resolved
```

把 `ParentDecisionId`、`Result`、`Rejected` 和 `PrototypeStatus` 返回给外层 `STAGE.md`；不要解释为全局 `Readiness`。外层编排器负责合并其他 decisions/open nodes，并决定是否继续设计或停止。

视觉-only 请求只有在外层确认没有其他 open node 时才可结束。进入生产实现时，保留结论，删除败者和切换器；原型代码不能未经正式验证直接升为生产代码。

## Anti-patterns

- 每个 UI 问题都强制做原型。
- 三个变体只差颜色或文案。
- 为“真实感”直接接生产写操作。
- 独立 HTML 假装能验证真实 auth、路由或数据密度。
- 集成式原型只用 `hostname === localhost` 防生产泄漏。
- 用户选完后仍强迫进入计划或实现。
